Class Link

Represents a connection to an ldap server

This class is an encapsulation of the Resource ldap_link, returned by php's ldap_connect() function and serves to simplify the configuration tasks of this link. You perform operations like connecting, binding, configuring the connection and reading the RootDSE entry using this object.

If you typecast this object into a string you will get the current domain name the link is connected to.

Method Chaining

Methods may be chained together to simplify configuration. Ensure that the methods you wish to chain together return the ADX\Core\Link object to avoid issues. See the example code below.

Serialisation support

You can serialise / unserialise the object to preserve the configuration across server requests. This allows you to configure a link with the needed features, perform an operation on the server, then store the link to i.e. a database for later re-use. If the user requests next operation, you can restore the object from the serialised data and continue using the link without any further configuration ( you will have to ADX\Core\Link::bind() again with the same credentials, though ).
This is especially useful when performing paged search results - you get the first page, store the link in database, present the data to user and if user requests the next page, you use the same link to retrieve next batch of results.

Make sure you import the class into current namespace before trying to use it:
use ADX\Core\Link;
To simplify the examples, the above line is not present in the examples but it is assumed you have it in your implementation.

Example:

$link = new Link( 'example.com' );      // Connect to example.com over standard port 389
// Typecasting to string gives you the domain name this link is connected to
echo "$link";   // example.com

// Chaining methods...
$link->use_tls()->bind( 'user@example.com', 'MySecretPwd' );

// Serialisation support
$serialised = serialize( $link );
// Save $serialised to a database...
// WARNING - $link is no longer usable after being serialised!

// In a later server request...
$link = unserialize( $serialised ); // Retrieve $serialised from your storage first
$link->bind( 'user@example.com', 'MySecretPwd' );       // Use the link with previously configured features

Create a new connection to a directory server

Create a new connection to a directory server

Connects to a directory server and reads the RootDSE entry ( via anonymous bind ). Note that ldap v3 is enforced for all operations and cannot be overriden.

Example

$link = new Link( 'example.com' );      // Uses default ldap port 389
$link = new Link( 'example.com', 636 ); // Uses ssl-specific port 636
$link = new Link( 'example.com', 389, [ServerControl::ShowDeleted] );   // Allows listing of deleted objects

Bind to the directory server using provided credentials ( or anonymously if none provided )

Bind to the directory server using provided credentials ( or anonymously if none provided )

This method performs the standard ldap_bind() operation on the ldap link. Before that, it performs the link configuration, like setting the protocol version and applying user-specific server controls.

Example

public ADX\Core\Link
# use_tls( )

Enable TLS security layer for this connection

Enable TLS security layer for this connection

Use this method to enable Transport Layer Security on the connection. You should use this before binding to the directory server, otherwise you risk compromising authentication data.

This method does not check if you have a valid certificate installed on the machine - it only invokes the standard ldap_start_tls() function - the host machine's configuration necessary for a successful TLS connection is up to you.

Example

$link = new Link( 'example.com' );      // Connect to server
$link->use_tls();                       // Enable TLS BEFORE binding
$link->bind( 'user@example.com', 'MySecretPassword' );  // Perform authenticated bind attempt

ADX\Core\Link

Show deleted objects in the search results

Show deleted objects in the search results

Invoke this method to inform the directory server that it should also include deleted objects in the search result. This is a shorthand way of doing the following:

use ADX\Enums;                          // Import the Enums namespace
$link = new Link( 'example.com' );      // Connect to server
$link->use_extended_control( Enums\ServerControl::ShowDeleted, false ); // Enable as non-critical feature

ADX\Core\Object that represents the RootDSE of a directory server.

ADX\Core\Object that represents the RootDSE of a directory server.

This information is generated automatically for each new Link object and the information is used across the library. You should not attempt to modify the object.

These attributes ( represented as ADX\Core\Attribute ) are available by default:

  • dnshostname
  • defaultnamingcontext
  • supportedcontrol
  • supportedldapversion
  • supportedsaslmechanisms
  • rootdomainnamingcontext
  • configurationnamingcontext
  • schemanamingcontext
  • namingcontexts
  • currenttime

Example

// Read the current time of the directory server
$link           = new Link( 'example.com' );
$server_time    = $link->rootDSE->currenttime->value( 0 );
